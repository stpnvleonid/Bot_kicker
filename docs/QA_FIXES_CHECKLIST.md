# QA чеклист после фиксов (proxy + planner + calendar sync)

Этот чеклист нужен для быстрой ручной проверки после деплоя.

## 1) Calendar sync (P0 SQL fix)

Цель: убедиться, что `Job1` не падает на фазе "missing events".

1. Запустить бот и подождать один проход `Job1` (или выполнить `sync_now`).
2. Проверить логи:
   - нет ошибок SQLite/bind параметров;
   - есть строки вида `Calendar ... fetched=..., inserted=..., updated=..., cancelled=...`.
3. Негативный сценарий:
   - отменить событие в календаре,
   - убедиться, что sync отрабатывает без SQL исключений.

## 2) Planner timezone and day consistency (P1)

Цель: все ключевые даты планера считаются по Москве.

1. В логах startup проверить, что Job5 10:00 и 20:00 идут по Moscow timezone.
2. Вечером открыть `/planner` и экран "Выполнено":
   - дата на экране соответствует московской дате.
3. После полуночи МСК (00:xx–01:xx) проверить:
   - `runPlannerExportIfAllUpdated` и `runPlannerFinalExport` работают с ожидаемой "закрываемой" датой.

## 3) Morning delivery exception (P1)

Цель: утренний инвайт в планер доходит даже при `notify_dm=0`.

1. Для тестового студента выставить:
   - `planner_enabled=1`,
   - `notify_dm=0`,
   - `dm_blocked=0`.
2. Запустить утренний сценарий (`/planner` или через job).
3. Проверить, что `planner_invite` в `send_queue` для этого студента создаётся и уходит.

## 4) Telegraf multi-proxy retry (P2)

Цель: при падении первого прокси бот пробует второй.

1. В `.env` указать 2+ прокси:
   - первый заведомо недоступный,
   - второй рабочий.
2. Перезапустить контейнер.
3. В логах ожидать:
   - `Telegraf: SOCKS agents attached (priority order): ...`
   - `getMe via SOCKS 1/N ...` (ошибка),
   - `getMe via SOCKS 2/N ...` (успех),
   - `Telegram OK @...`.

## 5) job_health upsert behavior (P2)

Цель: `last_success_at` не теряется после ошибки.

1. Зафиксировать успешный запуск job (например Job2).
2. Спровоцировать ошибку в этой job.
3. Проверить `job_health`:
   - `last_success_at` остаётся заполненным;
   - `last_error_at` и `last_error_message` обновились.

## 6) FIO flow moved to first `/start` only (P2)

Цель: ФИО не перезаписывается и не перехватывается в планере.

1. Новый пользователь:
   - нажать `/start`,
   - если Telegram-профиль без полного ФИО — получить одноразовый запрос `Фамилия Имя`,
   - отправить `Иванов Иван`,
   - проверить, что ФИО сохранилось в `students`.
2. Повторный `/start`:
   - ФИО в БД не меняется на данные профиля Telegram.
3. Planner flow:
   - при `planner_count_*` не должно быть отдельного режима "введите фамилию";
   - произвольный следующий текст не должен записываться в `last_name` автоматически.

## 7) Регрессии после фиксов

- `/start`, `/settings`, `/subjects`, `/planner` отвечают без ошибок.
- `Job2` и `send_queue_worker` продолжают работать как раньше.
- В логах нет новых `Fatal`/`Unhandled` ошибок.
