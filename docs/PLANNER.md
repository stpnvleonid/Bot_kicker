## Планер учебных задач

Документ включает:
- экспорт планера в Google Sheets
- сценарий планирования на неделю (`/planner_week`)

---

## 1. Интеграция с Google Sheets

### 1.1. Что делает интеграция

- Планер хранит задачи в БД (`daily_tasks`).
- Бот периодически **пересчитывает отчёт за день** и пишет его в Google‑таблицу:
  - каждые 2 часа (по Москве);
  - при раннем экспорте в окне 20:00–02:00, если нет задач в статусе `planned`;
  - финально в 02:00 (за вчера);
  - по команде админа `/planner_export_now [YYYY-MM-DD]`.
- Таблица — **отчёт поверх БД**, не источник истины.

### 1.2. Настройка Google Cloud

1. Включи в проекте Google Cloud:
   - **Google Calendar API**
   - **Google Sheets API**
2. Создай Service Account и JSON‑ключ (если ещё нет).
3. Сохрани JSON‑файл и пропиши путь в `.env`:

```env
GOOGLE_APPLICATION_CREDENTIALS=./secrets/calendar-service-account.json
```

### 1.3. Настройка таблицы

1. Создай таблицу в Google Sheets или используй существующую.
2. В первой строке задай заголовки (пример):
   - `A1`: `ФИО (ID)`
   - `B1`: `Предметы`
   - `C1`: `Выполнение`
   - дальше по 2 столбца на каждый будний день (по умолчанию в коде):
     - Пн: `D` (задачи), `E` (результат)
     - Вт: `F`, `G`
     - Ср: `H`, `I`
     - Чт: `J`, `K`
     - Пт: `L`, `M`
3. В колонке A для каждого студента задай строку в формате:
   - `Фамилия Имя Отчество (ID:123)`
   - или `Фамилия Имя (ID:123)`, если отчества нет.
4. Расшарь таблицу на сервисный аккаунт:
   - адрес из `client_email` в JSON‑ключе;
   - роль: **Редактор**.
5. Если включена защита листа/диапазонов — добавь сервисный аккаунт в исключения для колонок с задачами и результатами.

### 1.4. Переменные окружения

В `.env` должны быть:

```env
GOOGLE_APPLICATION_CREDENTIALS=./secrets/calendar-service-account.json
PLANNER_SHEET_ID=<ID_из_URL_таблицы>
PLANNER_SHEET_RANGE=Лист1!A:D
```

- `PLANNER_SHEET_ID` — строка между `/d/` и `/edit` в URL таблицы.
- `PLANNER_SHEET_RANGE` — лист и диапазон; колонка `A` используется как “ФИО (ID:...)”.

После изменения `.env` перезапусти бота.

### 1.5. Как бот находит строки студентов

- Для каждой строки `A2:A`:
  - парсится хвост `(...ID:123)` → `student_id`;
  - по `student_id` берётся ФИО из БД;
  - ФИО из БД сверяется с текстом перед скобками.
- Если ID или ФИО не совпадают:
  - строка **пропускается**, данные по этому студенту не пишутся;
  - всем пользователям из таблицы `admins` отправляется отчёт в Telegram с описанием проблемных строк.
- Если в строке нет `(ID:...)`, используется fallback по ФИО (как в старой версии), если это включено в коде.

### 1.6. Когда обновляется таблица

Бот делает **полный пересчёт дня**:
- каждые 2 часа (по Москве) — для текущей даты;
- в окне 20:00–02:00 (по Москве), если за дату нет задач в статусе `planned`;
- в 02:00 (по Москве) — для вчерашней даты;
- вручную по команде админа:

```text
/planner_export_now            # пересчитать отчёт за сегодня
/planner_export_now 2026-03-17 # пересчитать отчёт за указанную дату
```

При пересчёте:
- берутся все задачи за день со статусами `completed` / `partly_done`;
- по каждому студенту формируется список задач и статус дня (`Сделано` / `Частично` / `Не сделано`);
- соответствующие ячейки дня в таблице перезаписываются.

### 1.7. Ручной экспорт через npm (опционально)

Для первоначальной загрузки/отладки можно использовать скрипт:

```bash
npm run debug-planner-export
npm run debug-planner-export -- --date=2025-03-15
npm run debug-planner-export -- --all
```

Он вызывает старую логику выборочного экспорта и может быть полезен как запасной инструмент.

### 1.8. Краткий чеклист

- В Google Cloud включены **Google Calendar API** и **Google Sheets API**.
- Таблица создана, колонка A заполнена строками вида `Фамилия Имя (ID:123)`.
- Сервисный аккаунт (`client_email` из JSON‑ключа) добавлен в доступ к таблице с правами “Редактор”.
- В `.env` заданы `GOOGLE_APPLICATION_CREDENTIALS`, `PLANNER_SHEET_ID`, `PLANNER_SHEET_RANGE`.

### 1.9. Smoke-чеклист (обязательные exams + модерация)

- Утром (`/planner` → `Запланировать задачи на сегодня`) бот показывает блок обязательных `exams`: количество и короткие строки вида `<Предмет> Урок <DD.MM>` / `<Предмет> ДЗ <DD.MM>`.
- После выбора количества задач бот пишет итог на день: `X планерных + Y обязательных = Z`.
- На экране `Отметь выполненные задачи на сегодня (YYYY-MM-DD)` видны:
  - обычные задачи (если есть),
  - обязательные `exams` со статусом не `confirmed`,
  - кнопка `📸 Отправить фото` (или `↩ Фото заново` после отклонения).
- Студент отправляет фото; дату удобно указать **в подписи к фото** (`07 03`, `7 марта` и т.п.) — тогда один шаг. Иначе после фото отдельным сообщением в том же формате — запись уходит админам на модерацию.
- Админ нажимает `✅ Подтвердить` или `❌ Отклонить`:
  - при подтверждении студент получает `✅ Подтверждено`,
  - при отклонении — `❌ Отклонено` и может отправить evidence заново.
- Если обязательных больше 4, в экране показываются top-4, остальные подтягиваются в следующих напоминаниях.

### 1.10. ФИО и первый `/start`

- ФИО студента (`students.first_name`, `students.last_name`) больше не перезаписываются при повторном `/start`.
- Если это первый `/start` и в профиле Telegram нет полного ФИО, бот один раз просит отправить строку `Фамилия Имя`.
- Этот ввод нужен для корректного матчинга с Google Sheets (`ФИО (ID:...)`).
- Планер больше не трактует "первое произвольное сообщение как фамилию": шаг ФИО живёт только в потоке первого `/start`.

## 2. Планирование на неделю (`/planner_week`)

### 2.1. Цель

Дать студенту возможность заранее раскидать задачи по будням текущей недели, **не ломая** существующий сценарий «план на сегодня» и интеграцию с Google‑таблицей.

Основной UX остаётся дневным, недельное планирование — отдельный режим для предварительного ввода задач на будущие дни.

### 2.2. Пользовательский сценарий

#### 2.2.1. Вход в недельный режим

- Команда в личке: `/planner_week`.
- Ответ бота:
  - Заголовок: «План на неделю».
  - Текст: пояснение (что это за текущую/ближайшую неделю).
  - Кнопки дней:
    - `Пн`, `Вт`, `Ср`, `Чт`, `Пт` (только будни),
    - `Сегодня` (переход к текущему дню),
    - `Закрыть` (выйти).

#### 2.2.2. Экран выбранного дня

Для выбранного дня `task_date`:
- Заголовок: `Планы на YYYY-MM-DD`.
- Список задач за день с иконками статуса.
- Кнопки:
  - `Изменить i`, `Удалить i` для каждой задачи;
  - `Добавить задачу`, `Другой день`, `Сегодня`.

#### 2.2.3. Добавление / редактирование задач

Поведение реиспользует дневной планер:
- `Добавить задачу`: создаёт записи в `daily_tasks` для указанной даты.
- `Изменить i`: правит задачу для выбранной даты.
- `Удалить i`: удаляет запись и перерисовывает список.

Все действия остаются идемпотентными и работают только внутри выбранного дня.

### 2.3. Статусы и связь с джобами

Статусы задач:
- `planned` — запланирована,
- `completed` — выполнена,
- `partly_done` — частично выполнена,
- `cancelled` — отменена.

Недельное планирование не меняет бизнес‑логику: оно лишь создаёт/редактирует задачи в `daily_tasks` на разные даты.

Джобы планера (`Job5`) по-прежнему работают только с текущей датой, а недельные планы — это заранее созданные записи в `daily_tasks` для будущих `task_date`.
